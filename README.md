# LINE ワードカウンターBot

LINEのトークルーム(グループトーク・複数人トーク・個人トーク)で発言された
特定のワードの出現回数をカウントし、検出するたびに累計回数をトークルームに
返信するBotです。

## できること

- `.env` の `TARGET_WORDS` に設定したワードが発言に含まれるたびに検出し、
  そのトークルームでの累計出現回数を返信します。
- 1つのメッセージ内に同じワードが複数回含まれていても正しくカウントします。
- `!count ワード` と発言すると、返信を送らずに現在の累計回数だけを確認できます
  (ワードを省略すると `TARGET_WORDS` の先頭のワードが対象になります)。
- `!lately ワード` と発言すると、そのワードを直近に発言したユーザーの表示名を
  返信します(ワードを省略すると `TARGET_WORDS` の先頭のワードが対象になります)。
  **`!count` / `!lately` で参照できるのは `TARGET_WORDS` に設定したワードのみ**です
  (Botはそれ以外の単語の発言履歴を記録していないため)。
- `!reset ワード` と発言すると、そのワードのカウント(このトークルームのみ)を
  0にリセットします。ワードは省略できません(誤操作防止のため、省略時は
  使い方の案内を返信するだけで何もリセットしません)。
- `!all_reset` と発言すると、このトークルームの全ワードのカウントを
  一括で0にリセットします。
- `!stop` と発言すると、このトークルームでのワード検出を一時停止します。
  (再開するまでワードが発言されてもカウントされません)
- `!start` と発言すると、一時停止していたワード検出を再開します。
- `!state` と発言すると、このトークルームでの検出が現在「稼働中」か
  「停止中」かを返信します。
- **毎日 12:00 と 24:00(=翌日0:00、日本時間)に、全てのトークルームの
  全ワードのカウントを自動的に0にリセット**します(`node-cron`による定期実行)。
  一時停止(`!stop`/`!start`)の状態には影響しません。
- カウントはトークルームごとに独立して記録されます(グループAとグループBの
  カウントは混ざりません)。`!stop`・`!start`・`!reset`・`!all_reset` も
  それぞれのトークルーム単位で独立して効きます。

### 管理コマンドについての注意点

- `!stop` / `!start` / `!reset` / `!all_reset` / `!state` は、そのグループの
  メンバーであれば誰でも実行できてしまいます。悪戯防止のため特定のユーザーのみに
  制限したい場合は、`event.source.userId` を管理者リストと突き合わせるチェックを
  `index.js` に追加してください。
- サーバーを再起動しても、定期リセットのスケジュール(毎日12:00/24:00)は
  サーバー起動中であれば自動的に再開されます。ただしサーバーが停止していた
  タイミングでリセット時刻を迎えた場合、そのリセットは実行されません
  (見送られた回のリセットを後から実行する機能は含まれていません)。

### `!lately` コマンドについての注意点

- 直近の発言者のユーザーIDを記録しておき、コマンド実行時に
  LINEのプロフィール取得API(`getGroupMemberProfile`など)を呼び出して
  現在の表示名を取得しています。そのため、そのユーザーがすでに
  グループ・複数人トークを退出している場合は表示名を取得できず、
  「(表示名を取得できませんでした。退出済みの可能性があります)」と
  返信されます。
- 個人トーク(Botと1対1)の場合は `getProfile` を使用しますが、
  相手がBotをブロックしている場合など、同様に取得に失敗することがあります。

## 事前準備 (LINE Developersコンソール)

1. [LINE Developersコンソール](https://developers.line.biz/console/) で
   プロバイダーを作成し、「Messaging API」チャネルを作成します。
2. 作成したチャネルの「Messaging API設定」タブで以下を設定します。
   - **チャネルアクセストークン(長期)**: 「発行する」から発行し、値を控えておく
   - **チャネルシークレット**: 「チャネル基本設定」タブに表示されている値を控えておく
   - **Webhookの利用**: オンにする
   - **応答メッセージ**: オフにする(LINE公式アカウントの自動応答とBotの返信が
     競合しないようにするため。LINE Official Account Manager側の「応答設定」で変更します)
   - **グループトーク・複数人トークへの参加を許可する**: グループ内で使う場合はオンにする
     (デフォルトは無効になっています)

## セットアップ

```bash
npm install
cp .env.example .env
# .env を開いて CHANNEL_ACCESS_TOKEN / CHANNEL_SECRET / TARGET_WORDS を設定
npm start
```

デフォルトではポート3000で起動します。

## ローカルでの動作確認 (ngrokなど)

LINEのWebhook URLはHTTPSである必要があるため、ローカルで試す場合は
[ngrok](https://ngrok.com/) などでトンネリングします。

```bash
ngrok http 3000
```

発行された `https://xxxx.ngrok-free.app` に `/webhook` を付けた
URL (`https://xxxx.ngrok-free.app/webhook`) を、LINE Developersコンソールの
「Messaging API設定」タブの「Webhook URL」に登録し、「検証」ボタンで
接続確認を行ってください。

## 本番運用時の注意点

- このサンプルはカウントを `data/counts.json` というローカルファイルに
  保存しています。常時起動しているサーバー(自前のVPS、Renderの永続ディスクなど)
  であれば問題ありませんが、**Vercel FunctionsやAWS Lambdaのような
  サーバーレス環境ではファイルシステムが実行のたびにリセットされる**ため、
  カウントが保持されません。その場合は `store.js` の中身を
  Redis・Firestore・DynamoDBなどの外部データストアを使う実装に
  差し替えてください(関数のインターフェースは変えずに済むように
  作ってあります)。
- Botを複数人が同時に使う環境では、`store.js` は同期的なファイルI/Oで
  読み書きしているため単純な競合は避けられますが、アクセス数が多い場合は
  データベースへの移行を推奨します。

## ファイル構成

```
line-word-counter-bot/
├── index.js         # Webhookサーバー本体
├── store.js         # カウントの永続化(JSONファイル)
├── package.json
├── .env.example     # 環境変数のサンプル
└── data/
    └── counts.json  # カウントデータ(自動生成されます)
```
